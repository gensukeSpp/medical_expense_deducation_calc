# Issue #20: Human-in-the-Loop 修正UIの実装

## 目的

抽出した必要データ（氏名、クリニック名、支払い金額、発行日）をユーザーが確認・修正できる Web UI を実装する。修正結果はデータベースと JSON ファイルの両方に反映し、後続のテンプレート補正値学習（次回タスク）の基盤とする。

## 参照

- [要件定義書.md](../../要件定義書.md) 「5. ロジックフロー: 3, 4」「6. 実装ステップ: ステップ4」
- [Issue #4 出力スキーマ](../../tasks/issue_4/structured_output_schema.json)

## スコープ（Phase 1）

### 含むもの
- FastAPI ベースの Web サーバーの立ち上げ
- htmx を用いたフロントエンド UI（トップページ・詳細編集ページ）
- `output_json/*-structured_data.json` の読み取りと一覧表示
- ユーザー修正時の DB（`corrections` + `receipts.normalized_json`）と JSON ファイルの両方への反映
- CLI `--serve` フラグによる Web サーバー起動

### 含まないもの（次回タスク）
- 修正値と座標の紐付け（`要件定義書.md` フロー5, ステップ4）
- テンプレート補正値（`templates.coords_corrections`）の自動更新
- 命名抽出LLMの実装（`RealLLMClient`）
- 認証・認可機能

## 受入条件

1. トップページで抽出データの一覧が表示されること
   - 表示名: `{クリニック名(薬局名)}-{日付}`（フォールバック: ファイル名）
2. 各項目をクリックすると詳細ページへ遷移すること
3. 詳細ページで「ラベル: 値」形式のデータが表示されること（座標は非表示）
   - ラベル名: 「氏名」「クリニック名(調剤薬局名)」「支払い金額」「発行日」
4. 「修正」ボタンクリック後、以下の両方が更新されること
   - DB: `corrections` テーブルへのレコード追加 + `receipts.normalized_json` の更新
   - JSON ファイル: `output_json/*-structured_data.json` の内容
5. htmx により修正後、ページ遷移なしで更新値が再表示されること
6. エラー時は処理が継続し、エラーログに記録されること
7. 受入テスト（モック使用）が全てパスすること

## 技術選定

| 項目 | 採用 | 理由 |
|------|------|------|
| Web フレームワーク | **FastAPI** | 軽量・高速、Jinja2 + htmx との相性良好 |
| ASGI サーバー | **uvicorn** | FastAPI 標準 |
| フロントエンド | **htmx** (CDN) | サーバーレンダリングHTMLと自然に結合、ビルド不要、修正フロー（POST → 部分更新）に最適 |
| テンプレートエンジン | **Jinja2** | FastAPI 標準（starlette 組み込み） |
| CSS | 独自簡易CSS or 最小限 | 特別なフレームワーク不要 |
| DB | SQLite（既存） | `app/db.py` の CRUD API を使用 |
